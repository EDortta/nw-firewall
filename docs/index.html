auth-monitor v5

Um ataque contra um nó
é um ataque contra toda a rede

auth-monitor v5 distribui inteligência de ameaças entre servidores em tempo real — quando qualquer nó detecta um IP malicioso, todos os outros bloqueiam antes de receber a primeira requisição.

O problema que resolve

Firewalls tradicionais protegem servidores individualmente. Um scanner inteligente não satura um único endpoint — ele rotaciona entre dezenas de alvos, colhendo informações abaixo de qualquer limiar individual de alerta. Cada servidor vê apenas uma fração do comportamento malicioso.

"Alguém na rede já viu esse IP se comportar mal?"

auth-monitor elimina esse ponto cego ao compartilhar inteligência de ameaças entre todos os nós em segundos. Um bloqueio em qualquer servidor propaga-se imediatamente para toda a infraestrutura — antes que o atacante tente o próximo alvo.

Defesa isolada

  • Cada servidor decide sozinho
  • Scanner rotaciona entre alvos
  • Padrão fragmentado não é detectado
  • Janela de exposição entre nós

Defesa coletiva

  • Detecção compartilhada em tempo real
  • Bloqueio simultâneo em toda a rede
  • Padrões distribuídos são correlacionados
  • Sem janela entre detecção e proteção

Como funciona

1

Detecção em camadas

O detector roda como oneshot (systemd timer, 1 min) e analisa apenas as linhas novas dos logs — nunca re-lê o arquivo inteiro. Três camadas de decisão:

  • Bloqueio imediato — qualquer requisição a caminhos zero-tolerância (.env, .git/config, wp-config.php, exploits conhecidos) aciona o bloqueio sem acumulação de pontos.
  • Janelas deslizantes — bursts de HTTP 404, falhas SSH, probing de caminhos de alto risco. Cada sinal tem peso e janela temporal persistida em SQLite.
  • Assinatura de scanners — user-agents conhecidos (Shodan, Zgrab, Masscan, Nuclei…) e path traversal identificados diretamente.

2

Aplicação local primeiro

O agente lê o outbox (SQLite WAL) e aplica o bloqueio no kernel via ipset antes de publicar qualquer evento na rede. Um nó sem conectividade MQTT continua protegendo-se. Ao reiniciar, reconcilia automaticamente o ipset com o estado do banco.

3

Propagação criptograficamente verificada

O evento é publicado via MQTT (TLS, QoS 1) com assinatura HMAC-SHA256 canônica. Cada receptor valida, nesta ordem: assinatura → frescor (±300 s) → anti-replay (event_id único) → Guard de IP (nunca bloqueia privado, self ou broker). Só então aplica o bloqueio.

4

Toda a rede reage em segundos

Todos os outros nós recebem, verificam e bloqueiam o IP via ipset antes que o atacante faça a próxima requisição. O bloqueio tem TTL configurável — o kernel expira automaticamente, sem cron ou processo extra.

Protocolo de eventos

Cada evento publicado no grid tem estrutura versionada e assinatura que cobre todos os campos (chaves ordenadas, sem espaços — replay de um campo modificado invalida a assinatura).

{
  "v": 5, // versão do protocolo
  "event": "block", // block | unblock | heartbeat | sync_state
  "event_id": "uuid4", // anti-replay: descartado se já visto
  "ts": "2026-06-11T05:37:00Z", // rejeitado se > ±300 s do receptor
  "node": "web-prod-01", // emissor
  "ip": "1.2.3.4",
  "reason": "ssh_auth_fail_burst",
  "ttl": 604800,
  "sig": "base64(HMAC-SHA256(canonical_json))"
}

O que mudou na v5

🔐

HMAC-SHA256 em todos os eventos

Eventos sem assinatura válida são descartados silenciosamente. Suporta rotação de chave sem downtime (chave anterior aceita por janela configurável).

🔁

Anti-replay persistente

Cada event_id é registrado em SQLite. Reenvio de um evento legítimo capturado não produz efeito — independente do TTL de frescor.

🛡

Guard de IP

Antes de qualquer bloqueio: o IP é público? Está na allowlist? É o broker? É o próprio nó? Um evento forjado não auto-bloqueia a infraestrutura.

🔒

TLS no broker

Mosquitto 2.x na porta 8883 com cert Let's Encrypt. Porta 1883 acessível apenas em localhost. Autenticação por usuário/senha por grupo de nós.

⚡

Local-first + reconciliação

O agente aplica o bloqueio no kernel antes de publicar. No boot, reconcilia o ipset com o banco — reboots não abrem janela de exposição.

🗄

SQLite WAL + decisões auditáveis

Toda ação (block, unblock, rate_limited, api_reject…) fica na tabela decisions com motivo e timestamp. Logtail por inode+offset, rotation-safe.

🚫

Segredos fora do config

Se password, hmac_secret ou api_key aparecerem no config.json, o processo recusa inicializar. Segredos só via variáveis de ambiente.

📊

Rate limiter

Janela deslizante de 30 bloqueios/min. Proteção contra mass-block acidental ou forjado — bursts de startup não propagam ruído para toda a rede.
